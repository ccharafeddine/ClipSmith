2026-07-02 — v2, output format picker (ClipSmith as a converter)
----------------------------------------------------------------
User request: keep MP4 default, but let the export write other video file
types — effectively a built-in container/codec converter. This relaxes the old
"always mp4" hard constraint (updated in CLAUDE.md).

Design: the trim + reframe filtergraph are codec-agnostic (yuv420p handoff), so
a format only changes the tail of the ffmpeg command (video codec, audio codec,
muxer). Two families:
  - MP4 / MOV / MKV = H.264 + AAC. Reuse the ENTIRE pipeline incl. the hardware/
    libx264 encoder choice. Always available (libx264 ships).
  - WebM = VP9 + Opus. Different codec path; hardware H.264 doesn't apply (VP9 is
    software libvpx-vp9). Needs libvpx/libopus in the bundled ffmpeg, which the
    from-source macOS build may lack — so WebM is RUNTIME-PROBED (VP9+Opus test
    encode) and only offered when it actually encodes. Windows BtbN build has it.

DONE
- src-tauri/src/formats.rs (new): OutputFormat enum (Mp4/Mov/Mkv/Webm),
  encode_args(encoder, out) building video+audio+container+muxer args (H.264
  defers to VideoEncoder; WebM = VP9 crf 31 / Opus 128k). probe_webm() test
  encode + available_cached() via a Tauri-managed OnceCell. faststart only for
  mp4/mov; forced -f muxer so a renamed output can't confuse detection.
- cutter.rs: cut() now takes a cutter::Encoding { format, encoder } (bundled to
  stay under clippy's 7-arg limit); tail built from format.encode_args().
- commands.rs: export_clip takes a grouped ExportOptions { reframe, format,
  useHardware }; new available_formats command. HW encoder only detected for
  H.264 formats.
- lib.rs: mod formats; manage FormatCache; register available_formats.
- Frontend: src/formats.ts (display metadata + formatInfo). state: outputFormat
  (default "mp4") + availableFormatIds signals, refreshFormats() on load, save
  dialog uses the chosen ext, exportClip passes the format. ipc: exportClip
  nests options + availableFormats(). ExportPanel: new Format field (chips over
  available formats + codec hint); Encoder field now shown only for H.264
  formats (WebM = VP9/Opus, no HW toggle).

2026-07-02 — v2 Reframe, Tier 1 Step 3 (hardware encoders)
---------------------------------------------------------
Goal: use a hardware H.264 encoder when the machine has one — h264_videotoolbox
(macOS) or h264_nvenc/h264_qsv/h264_amf (Windows) — with a clean libx264
fallback. Reframe (blur especially) + the Tier 2 analysis pass make software-only
encoding hurt. Completes Tier 1.

Key decision: detect by TEST-ENCODE, not by `ffmpeg -encoders`. Being listed in
the build != usable (nvenc listed on every Windows build but needs an NVIDIA
GPU). We encode 5 frames of testsrc with the exact codec+quality flags to null;
accept only on exit 0. Any failure cascades to the next candidate, then libx264.
So correctness is guaranteed — worst case is software. Cannot validate real GPUs
in this env; the fallback is what makes that safe. HW quality flags flagged for
live/CI tuning (MAINTAINER NOTE in encoders.rs).

DONE
- src-tauri/src/encoders.rs (new):
    * VideoEncoder enum { X264, VideoToolbox, Nvenc, Qsv, Amf }; per-encoder
      video_args() (codec + constant-quality knob + pix_fmt, tuned ~CRF 18):
      x264 -crf 18 / videotoolbox -q:v 60 / nvenc -rc vbr -cq 19 / qsv
      -global_quality 20 (nv12) / amf -rc cqp -qp 20. display_name() for the UI.
    * platform CANDIDATES (macOS: videotoolbox; Windows: nvenc,qsv,amf).
    * probe() 5-frame testsrc -> null encode; detect() first working candidate
      else X264; detect_cached() via a Tauri-managed tokio OnceCell (probe once).
    * #[allow(dead_code)] on the enum: variants are platform-gated construction.
- cutter.rs: cut() takes a VideoEncoder; the hardcoded libx264 block replaced by
  encoder.video_args() (pix_fmt moved into it). Audio/container flags unchanged.
- commands.rs: export_clip gains use_hardware (resolve encoder or force X264);
  new detect_encoder command returns the display name.
- lib.rs: mod encoders; manage EncoderCache; register detect_encoder.
- Frontend: hwAccel (Auto, default) + encoderLabel signals; refreshEncoder()
  called on load; exportClip passes hwAccel(); ipc detectEncoder(); ExportPanel
  gets an Auto/Software chip toggle + the resolved encoder name.

2026-07-02 — v2 Reframe, Tier 1 Step 1 (blur-fill + pad + Freeform crop)
------------------------------------------------------------------------
Goal: replace v1's destructive crop with a Reframe model — pick an output
canvas ratio, then a fill strategy for how the source fills it. This step ships
blur-fill and pad; crop-to-fill-with-scaling (Step 2) and hardware encoders
(Step 3) are deferred. v1 crop survives as the "Freeform" canvas.

Design decision (asked + answered): keep a free-crop escape hatch. Presets are
AR-locked; "Freeform" = unconstrained crop, output == the rectangle. That path
is literally v1's `crop=w:h:x:y` filter, so v1 crop is byte-preserved.

DONE
- Backend (src-tauri/src/cutter.rs): replaced the `Crop`-only path with a
  `Reframe { canvas_w, canvas_h, strategy, anchor, pad_color, crop }` struct

    * Pad  -> `-vf scale=…:force_original_aspect_ratio=decrease,pad=…:color`
    * Blur -> `-filter_complex` split/gblur/overlay graph, mapped as `[v]`
    * Crop -> `crop=…` (+ `,scale` only when canvas != crop dims). Freeform
      (canvas == crop dims) emits the bare `crop=` == v1 verbatim.
  cut() now branches None / Simple(-vf) / Complex(-filter_complex) and picks the
  matching `-map` (0:v:0 vs [v]). Everything else (frame-accurate -ss/-t, audio
  map, CRF 18, faststart, time= progress parse) unchanged.
- commands.rs: export_clip now takes `reframe: Option<cutter::Reframe>`.
